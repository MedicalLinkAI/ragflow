## 1. 模块概述

`qwen30b_ocr.py` 是基于 **Qwen3-VL-30B-A3B-Instruct** 多模态大模型的医疗文档 OCR 处理器，替代原有的 `qwen-vl-ocr` 方案，在 RAGFlow pipeline 中作为 Extractor 的后端执行。

**核心能力**：
- 文本类文档：3步流水线，提取全量文字 -> LLM 结构化提取 -> 精确坐标
- 表格类文档（LabReport）：3步流水线，提取检验项latex  -> LLM 结构化提取 -> 精确坐标

**部署形态**：本地 vLLM 推理服务（OpenAI-compatible API），支持 DashScope 云端回退。

## 3. 双流水线详细设计

### 3.1 process_table — LabReport 表格提取（3步流水线）

针对检验报告表格的特殊优化流水线，每页独立执行 3 步：

```
┌────────────────────────────────────────────────────────────────────┐
│ Step 0: PDF → 图片渲染 (200 DPI)                                    │
│   - 多页时裁剪到 positions 最大包围框                                 │
│   - 单页时使用全页图片                                                │
└────────────────────────────────┬───────────────────────────────────┘
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ Step 1: Image → LaTeX tabular  (qwen3-vl-30b)                      │
│   - 使用 TABLE_TO_LATEX_PROMPT                                     │
│   - 模型直接输出 LaTeX 代码，确保列数完整、不丢列                       │
└────────────────────────────────┬───────────────────────────────────┘
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ Step 2: LaTeX → JSON  (pipeline LLM, ext._generate_async)          │
│   - 复用 Extractor 原有的 sys_prompt + msg 构建逻辑                   │
│   - 从 LaTeX 精确提取结构化字段: name, item_code, value, unit...     │
└────────────────────────────────┬───────────────────────────────────┘
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ Step 3: Image + item_names → coordinates  (qwen3-vl-30b)           │
│   - 用已知检验项名称列表构建定位 prompt                                │
│   - 模型返回每个名称的 bbox [x1,y1,x2,y2] (0-1000归一化)              │
│   - 坐标映射回 PDF 页面坐标系 (考虑裁剪偏移)                           │
└────────────────────────────────────────────────────────────────────┘
                                 ▼
                     输出: row_positions + HTML 表格
```

**设计动机**：直接用模型提取 JSON 字段容易丢列、错位。LaTeX 中间表示强制模型"看到"每列每行，再交给传统 LLM 做结构化，两阶段职责分离，质量显著提升。

### 3.2 process_text — 非 LabReport 文本提取（3步流水线）

适用于门诊病历、处方单、购药记录等非表格类文档：

```
┌────────────────────────────────────────────────────────────────────┐
│ Step 0: PDF → 图片渲染 (200 DPI) + 裁剪                             │
│   - 多页 / PrescriptionRecord / MedicationRecord 时裁剪              │
│   - 单页其他类型时使用全页                                             │
└────────────────────────────────┬───────────────────────────────────┘
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ Step 1: Image → text array  (qwen3-vl-30b, TEXT_ONLY_PROMPT)       │
│   - 只识别文本，不返回坐标（减少模型负担，提高准确率）                    │
│   - 过滤纯符号行 ("++", "--" 等)                                     │
│   - 检测并截断模型重复输出（幻觉去重）                                  │
└────────────────────────────────┬───────────────────────────────────┘
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ Step 2: 文本拼接 → LLM 结构化提取                                  │
│   - 多页文本合并为 ocr_assembled_text                                │
│   - 复用 Extractor sys_prompt 做结构化提取                           │
│   - 支持 encounter_date 回写到 classify_result_tks                   │
└────────────────────────────────┬───────────────────────────────────┘
                                 ▼
┌────────────────────────────────────────────────────────────────────┐
│ Step 3: 坐标定位  (qwen3-vl-30b, _build_coord_prompt)              │
│   - 将已知文本行数组发送给模型，要求定位每行的 bbox                      │
│   - 未匹配的 bbox 用平均行高插值填充                                   │
│   - 坐标映射回 PDF 坐标系 (考虑裁剪偏移)                               │
└────────────────────────────────────────────────────────────────────┘
                                 ▼
                     输出: positions + 结构化 JSON
```

**设计动机**：文本和坐标分两步走。第一步只要求模型"读对字"，第二步只要求模型"找对位置"，降低单次任务复杂度，两阶段各自质量更可控。

## 4. 坐标映射体系

### 4.1 模型坐标 → PDF 坐标

模型输出 bbox 归一化到 `[0, 1000]`，映射公式：

```
pdf_x = model_x × (crop_w / 1000) 
pdf_y = model_y × (crop_h / 1000) 
```

### 4.2 坐标存储格式

| 字段 | 格式 | 页码索引 | 说明 |
|------|------|---------|------|
| `row_positions` | `[page_num, x0, x1, top, bottom]` | **1-indexed** | 表格流水线，每检验项一行 |
| `positions` | `[page_num, x0, x1, top, bottom]` | **0-indexed** | 文本流水线，每文本行一条 |

### 4.3 坐标插值填充（文本流水线）

当模型未返回某行 bbox 时，使用相邻已匹配行的**平均行高**进行插值：

```
向前查找最近已匹配行 → 用 avg_h × gap 推算 top/bottom
向后查找最近已匹配行 → 反向推算
```

确保 `positions` 数组与文本行 **1:1 对齐**，未匹配行用 `[pn, 0, 0, 0, 0]` 占位。

## 5. Prompt 工程

### 5.1 Prompt 矩阵

| Prompt | 用途 | 输出格式 | max_tokens |
|--------|------|---------|------------|
| `TEXT_ONLY_PROMPT` | 纯文本识别（Step3） | `["text1", "text2", ...]` | 16384 |
| `TABLE_TO_LATEX_PROMPT` | 表格→LaTeX（StepA） | LaTeX tabular 代码 | 16384 |

| `_build_table_prompt()` | 检验项坐标定位（StepC） | `[{text, bbox}]` | 8192 |
| `_build_coord_prompt()` | 文本行坐标定位（Step7） | `[{text, bbox}]` | 8192 |


### 6.2 Prompt 设计原则

1. **明确负面示例**：同时给出正确和错误示例，减少格式偏离
2. **字段严格约束**：`bbox` 必须是 4 元素数组，`text` 必须与原文完全一致
3. **禁止 Markdown 包裹**：明确要求直接输出纯 JSON/LaTeX
4. **医疗领域特化**：提示保留 ↑↓ 箭头、H/L 标识、★ 符号等特殊字符

## 7. API 调用层

### 7.1 三种调用模式

| 函数 | 返回值 | 适用场景 |
|------|--------|---------|
| `_call_qwen30b_text_only()` | `(data, elapsed, status)` | 用于非检验报告提取纯json数组文本 |
| `_call_qwen30b_latex_only()` | `(content, elapsed, status)` | 用于检验报告提取纯原始文本（LaTeX），不做 JSON 解析 |
| `_call_qwen30b_to_coord()` | `(items, elapsed, status)` | 用于获取坐标 |


### 7.2 配置参数

```python
API_ENDPOINT = os.environ.get("QWEN30B_OCR_API_ENDPOINT", "http://10.16.3.16:8090/v1/chat/completions")
MODEL_NAME   = os.environ.get("QWEN30B_OCR_MODEL", "Qwen/Qwen3-VL-30B-A3B-Instruct-FP8")
# temperature = 0.1 (固定，确保输出确定性)
# timeout = 120s
```

### 7.3 JSON 解析容错

```
严格解析 (json.loads)
    ↓ 失败
json_repair 修复解析
    ↓ 失败
返回错误状态，不抛异常
```

支持两种输出格式：
- `{"items": [...]}` — 对象包裹
- `[...]` — 直接数组

## 8. 鲁棒性设计

| 场景 | 处理策略 |
|------|---------|
| 模型返回空内容 | 跳过该页，日志记录，继续处理其他页 |
| JSON 解析失败 | `json_repair` 回退，仍失败则跳过 |
| 模型重复输出（幻觉） | 周期检测 + 截断（文本流水线） |
| 纯符号行 (`++`, `--`) | 正则过滤，不进入后续流程 |
| 坐标未匹配 | 插值填充或 `[0,0,0,0,0]` 占位 |
| 裁剪 bbox 无效 | 回退使用全页图片 |
| API 超时/HTTP 错误 | 返回错误状态，不重试（由上层 pipeline 处理） |

## 10. 三模型 OCR 能力对比与选型
结论
识别能力三者相近，都能完成文字提取任务
Qwen3-VL-30B-A3B-Instruct-FP8 综合最优：识别 + 定位都可靠
Qwen3.6-35B-A3B 定位可用但弱于30B，倾斜图片上定位精度下降
Qwen3-VL-32B-Instruct 定位不可用，坐标未归一化，无法用于下游区域裁剪等依赖坐标的场景
推荐方案：优先使用 30B 作为 OCR 主力模型；35B 可作为备选；32B 仅用于不需要坐标的场景。 综合最优：识别 + 定位都可靠


### 10.1 定位能力差异根因分析

#### Qwen3-VL-32B-Instruct — 定位不可用

**根因：架构代差**

32B 基于 **Qwen2.5-VL 架构**（底层视觉编码器沿用上一代设计），坐标输出习惯与 Qwen3 系列不同：

- 输出**像素级绝对坐标**（基于原图分辨率），而非归一化 0-1000 相对坐标
- 例：4959×7017 的图片，y 坐标可达 7000+，远超 0-1000 范围
- 虽然后处理可做 clip + 缩放校准，但精度损失较大，**倾斜图片上误差尤为明显**

#### Qwen3.6-35B-A3B — 定位可用但稍弱

**根因：纯文本模型 + 视觉适配层**

35B 是 **Qwen3.6 系列的 MoE 模型**（35B 总参数 / 3B 激活参数），本身**不是原生视觉语言模型**：

- 坐标归一化到 0-1000 正确（Qwen3 系列统一行为）
- 但视觉理解通过适配层接入，深度不如原生 VL 模型
- MoE 稀疏激活（3B active）在精细空间定位任务上参数容量不足
- **倾斜图片**上几何变换能力弱，定位精度明显下降

#### Qwen3-VL-30B-A3B-Instruct-FP8 — 定位最优

**根因：原生 VL + 最新架构**

30B 是 **Qwen3-VL 系列原生视觉语言模型**，同样采用 MoE 架构（30B 总 / 3B 激活），关键优势：

- 视觉编码器与语言模型**联合训练**，空间定位能力更强
- 坐标归一化经过**对齐训练**，严格遵守 0-1000 范围
- FP8 量化对坐标回归影响极小（量化主要影响生成质量）
- 倾斜图片的几何理解得益于原生视觉编码器的**旋转不变性训练**

### 10.2 架构对比总览

| 维度 | 32B | 35B | 30B |
|------|-----|-----|-----|
| **底层架构** | Qwen2.5-VL（上一代） | Qwen3.6 文本模型 + 视觉适配 | Qwen3-VL 原生多模态 |
| **坐标体系** | 像素绝对坐标（未归一化） | 0-1000 归一化 | 0-1000 归一化 |
| **空间理解** | 弱（架构老旧） | 中（非原生 VL） | 强（原生 VL 联合训练） |
| **倾斜鲁棒性** | 差 | 一般 | 好 |
| **推荐用途** | 不需要坐标的场景 | 备选 | **主力 OCR 模型** |

### 10.3 选型结论

> **Qwen3-VL-30B-A3B-Instruct-FP8** 作为项目主力 OCR 模型，识别能力最强、定位精度最高、倾斜鲁棒性最好，完全满足 3 步表格流水线和 2 阶段文本流水线的全部需求。

## 11. 关键设计决策记录

| 决策 | 原因 |
|------|------|
| LaTeX 中间表示 | 直接 JSON 提取丢列严重，LaTeX 强制模型逐列输出 |
| 文本和坐标分离 | 降低单次任务复杂度，两阶段各自质量更可控 |
| temperature=0.1 | 医疗数据要求确定性输出，避免随机性 |
| 1-indexed row_positions | 与前端高亮组件约定一致 |
| json_repair 回退 | qwen3-vl-30b 偶尔输出非标准 JSON |
